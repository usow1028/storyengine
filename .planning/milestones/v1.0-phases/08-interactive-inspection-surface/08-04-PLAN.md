<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inspection API DTO -> React state | Structured verdict, evidence, repair, and diff data become component props and state. |
| story/evidence text -> DOM | User-authored story-derived summaries render in the browser. |
| advisory/repair labels -> user interpretation | UI hierarchy can imply authority if hard and soft concepts are mixed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-14 | Tampering / XSS | `EvidenceSummary` and repair text rendering | mitigate | Render all story/evidence/repair strings as React text nodes; block `dangerouslySetInnerHTML` and `innerHTML`. |
| T-08-15 | Spoofing | `VerdictDetailPanel` hard verdict section | mitigate | Always render `Deterministic Verdict` before repairs/advisory content and derive verdict badges only from API `verdictKind`, never advisory scores. |
| T-08-16 | Repudiation | user-visible verdict rows | mitigate | Show exact verdict kind, finding ID/reason code summaries, and selected state so each visible judgment maps back to the DTO. |
| T-08-17 | Denial of Service | triage/detail layout | mitigate | Keep run-scoped rendering, stable dimensions, responsive stacking, and no graph/canvas primary surface in this plan. |
| T-08-18 | Information Disclosure | browser UI metadata | mitigate | Do not render `snapshotDir`, `snapshotSet`, `genreWeights`, `worldProfile`, raw prior artifacts, or raw storage rows. |
</threat_model>
