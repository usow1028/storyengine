<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| check request -> scope resolution | Untrusted `scopeId` selects stored draft scope and therefore determines which approved segments may participate in execution. |
| scope resolution -> verdict-run persistence | The resolved scope metadata becomes durable comparison identity for later diffing. |
| full-graph evaluation -> retained verdict set | The runner must preserve contextual reasoning but only persist verdicts anchored to the requested scope. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Tampering | scoped check eligibility | mitigate | Reject scoped execution unless every segment in the requested scope is approved and current. |
| T-11-02 | Integrity | cross-revision scope identity | mitigate | Persist deterministic `comparisonScopeKey` alongside raw `scopeId` and reject heuristic fallback. |
| T-11-04 | Integrity | scoped verdict persistence | mitigate | Keep full-graph evaluation for context but persist only verdicts anchored to in-scope event IDs. |
</threat_model>
