## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| submit request -> domain schemas | Untrusted draft metadata and source offsets enter Zod parsing. |
| raw draft text -> source refs | Raw text normalization determines later source-span traceability. |
| domain contracts -> future storage/API | New draft/scope objects will be persisted and serialized by later plans. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Tampering | `src/domain/drafts.ts` | mitigate | Validate draft IDs, section shape, source refs, and scope union with Zod before any storage/API use. |
| T-09-02 | Repudiation | `normalizeDraftSourceText` and `DraftSourceTextRefSchema` | mitigate | Store `textNormalization: "lf"` and require CRLF source-slice tests proving offsets match normalized text. |
| T-09-03 | Denial of Service / Tampering | SQL persistence | transfer | Plan 09-02 owns additive nullable SQL and legacy-row loading; this plan creates only schemas and pure planner output. |
| T-09-04 | Tampering | API response compatibility | transfer | Plan 09-03 owns additive API request/response contracts; this plan preserves the existing `segmentSubmissionText` signature. |
</threat_model>
