<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request -> submit schema | Untrusted draft metadata enters API parsing. |
| persisted snapshot -> HTTP response | Stored draft/scope/source metadata is serialized to clients. |
| existing clients -> extended API | Old clients must keep working against additive response fields. |
| approved ingestion -> check route | Existing check execution must remain explicit and not infer Phase 9 scopes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Tampering | `SubmitIngestionRequestSchema` and `serializeIngestionSessionResponse` | mitigate | Parse optional draft metadata with Zod schemas and parse serialized responses before sending. |
| T-09-02 | Repudiation | API `sourceTextRef` fields | mitigate | Serialize `sourceTextRef.textNormalization = "lf"`, offsets, and segment text together; API tests assert source refs survive round trip. |
| T-09-03 | Denial of Service / Tampering | existing ingestion clients | mitigate | Keep `chunk`, `full_draft`, `draftTitle`, `storyId`, `revisionId`, and old segment fields valid; test legacy chunk check flow. |
| T-09-04 | Tampering | API response contract | mitigate | Add fields only; do not rename/remove `sessionId`, `workflowState`, `storyId`, `revisionId`, `segments`, or candidate fields. |
</threat_model>
