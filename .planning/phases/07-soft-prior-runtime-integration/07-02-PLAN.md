<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API request -> server dependency config | Users submit/check ingestion sessions, but prior snapshot config must stay outside request payloads. |
| service result -> API schema | Zod response parsing decides what clients can inspect; unknown soft-prior fields would otherwise be stripped. |
| advisory evidence -> external response | The API should expose explainable assessment data without leaking raw prior artifacts or snapshot internals. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Tampering / Information Disclosure | Fastify check route and schemas | mitigate | Do not add `snapshotDir`, `snapshotSet`, `genreWeights`, or `worldProfile` to request schemas; only `StoryGraphApiDependencies.softPriorConfig` may carry prior config. |
| T-07-07 | Spoofing / Tampering | API check response | mitigate | Place all advisory output under a separate `softPrior` block and preserve top-level hard fields unchanged. |
| T-07-08 | Information Disclosure | `softPrior.assessment.contributions` | mitigate | Expose D-07 fields from `SoftPriorAssessmentSchema` and `RepairPlausibilityAdjustmentSchema`; do not expose raw `snapshotSet`, raw artifacts, or `sourceWorkIds`. |
| T-07-09 | Denial of Service | missing/disabled snapshot config | mitigate | Service tests must prove `status: "disabled"` and `status: "missing_snapshot"` still return `workflowState: "checked"` and a persisted hard run. |
| T-07-10 | Tampering | Zod response parsing | mitigate | Add `SoftPriorAdvisoryResponseSchema` to `CheckIngestionResponseSchema` so the route parse step retains the advisory block explicitly. |
</threat_model>
