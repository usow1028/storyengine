| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline metadata gaps, status-dot offset |
| sm | 8px | Compact control padding, chip gaps, trace-row gaps |
| md | 16px | Default component padding, verdict row spacing |
| lg | 24px | Panel padding, timeline item spacing |
| xl | 32px | Split-view gutters, major detail section spacing |
| 2xl | 48px | Empty-state block spacing, major vertical breaks |
| 3xl | 64px | Page-level top and bottom spacing only when viewport height permits |

## Typography

Use exactly four type sizes and two font weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Additional typography rules:

- Body copy, evidence summaries, repair descriptions, and advisory text use 16px/400/1.5.
- Metadata labels, verdict kind labels, chips, timestamps, and trace field names use 14px/600/1.4.
- Panel headings use 20px/600/1.2.
- Page title uses 28px/600/1.2.
- Do not use viewport-scaled font sizes, negative letter spacing, condensed fonts, or more than these two weights.

- Text contrast must meet WCAG AA: 4.5:1 for normal text and 3:1 for large text or non-text UI indicators.

## Layout Contract

First screen: the inspection console itself. Do not create a marketing landing page, hero section, feature page, or onboarding splash.

Desktop layout:

| Region | Contract |
|--------|----------|
| App shell | Full browser viewport minimum height, dominant background, no decorative frame around the primary experience |
| Header | 64px high, page title on the left, run metadata on the right |
| Left triage rail | 360px default width, 320px min, 420px max, secondary background, sticky within viewport |
| Right detail panel | Flexible width, minimum 0, dominant background, vertical scroll independent of triage rail |
| Split gap | 24px on desktop, 16px on compact desktop |

Responsive layout:

| Width | Contract |
|-------|----------|
| 1024px and wider | Two-column split view: triage left, detail right |
| 861px to 1023px | Two-column split view with 320px triage rail and 16px gap |
| 860px and narrower | Triage appears first, selected detail follows; keep current verdict summary sticky at top of detail while scrolling |
| 480px and narrower | Verdict rows stack metadata below title; no horizontal scrolling for text content |

Structural rules:

- Use full-width panels and bands, not decorative cards inside cards.
- Verdict rows may use a single 8px radius maximum and 1px border.
- Detail sections are separated by headings, borders, and spacing, not nested card frames.
- The timeline is a vertical ordered list for Phase 08. Do not make a graph-first canvas the primary interaction.

## Interaction Contract

Verdict grouping:

- The triage list order is fixed: `Hard Contradiction`, `Repairable Gap`, `Soft Drift`, `Consistent`.
- Groups with zero verdicts remain visible with count `0` so users can see that the category was checked.
- Default selection is the first verdict in the first non-empty group using the fixed group order.
- Selecting a verdict updates only the right detail panel and preserves the triage rail position.

Verdict detail:

- The top of detail always states the selected verdict kind and `Deterministic Verdict` before any repair or advisory content.
- Evidence summary appears before structured trace fields.
- Trace fields are collapsed by default and expand in place.
- Trace expansion must include available `findingId`, `reasonCode`, conflict path, missing premises, supporting findings, event IDs, state boundary IDs, and rule version IDs.
- Long trace arrays collapse after 6 visible rows with a `Show More Trace Fields` control.

Timeline:

- Timeline items are ordered by event sequence, not by render insertion order.
- Each item displays event summary, event ID, linked state boundary IDs, and linked rule version IDs when present.
- Timeline must not infer new reasoning. It visualizes evidence already present in the inspection DTO.

Repairs:

- Repair candidates appear inside the selected verdict detail after evidence and timeline.
- Repairs display rank, repair kind, writer-readable repair direction, and plausibility adjustment when present.
- Do not provide an apply, rewrite, accept, or auto-fix button in Phase 08.

Soft-prior advisory:

- Soft-prior advisory is a separate band with heading `Advisory Pattern Signal`.
- The advisory band must not be nested inside `Deterministic Verdict`, `Evidence Summary`, or `Repair Candidates`.
- Available advisory data displays status, dominant layer, representative pattern summary, triggered drifts, contribution evidence, repair plausibility adjustments, and unavailable-state reasons where applicable.
- The advisory band always includes: `Pattern signal only. Hard verdict remains deterministic.`
- Advisory values must never change the selected verdict kind, group count, status badge, or deterministic evidence summary.

Run and diff context:

- Run metadata may appear in the header or a compact detail metadata row.
- Previous-run diff data is secondary. It may appear as small badges or an expandable section, but it must not replace verdict triage as the starting point.

## Data Boundary Contract

Hard and soft data must be structurally separate in the API view model and the UI component tree.

| Data Class | UI Location | Boundary Rule |
|------------|-------------|---------------|
| `verdictKind` | Triage group, verdict row, deterministic verdict header | Authoritative deterministic classification |
| Evidence summaries | Evidence summary and timeline | Writer-readable first, structured trace on expansion |
| Repair candidates | Repair section inside selected verdict detail | Suggestions only; no automatic application |
| `softPrior` advisory | Separate advisory band | Advisory only; cannot override hard verdict truth |
| Raw prior artifacts | nowhere | Do not expose `sourceWorkIds`, `snapshotDir`, `snapshotSet`, raw corpus rows, or request-supplied prior config |

Security and rendering rules:

- Render story text, event summaries, evidence strings, and advisory summaries as React text nodes.
- Do not use `dangerouslySetInnerHTML`.
- Do not accept snapshot paths, genre weights, prior config, or world profile fields from browser requests.
- API errors must not expose filesystem paths, raw storage rows, stack traces, or prior artifact internals.

---

## Accessibility Contract

- Use semantic headings in visual order: page title, triage, selected detail, section headings.
- Verdict rows are buttons or links with visible focus and `aria-current="true"` on the selected verdict.
- Group expanders use `aria-expanded` and preserve keyboard focus after toggling.
- Trace and advisory expand controls must be reachable by keyboard and have a 44px minimum hit target.
- Keyboard order follows header, triage groups, selected verdict detail, expandable trace controls.
- The active verdict must be distinguishable by text, border, and focus/selected state, not color alone.
- Loading, error, and empty states must use live regions only if content changes after initial page load.

## Out-of-Scope Guardrails

Do not implement these in Phase 08:

- Marketing landing page or hero.
- Full graph-first visualization, pan/zoom canvas, or graph layout library as the primary surface.
- Automatic repair application, rewrite generation, or accept/apply buttons.
- Raw prior browsing, corpus work browsing, or similar-case exploration.
- Collaboration, export, comments, sharing, or manuscript-scale navigation.
