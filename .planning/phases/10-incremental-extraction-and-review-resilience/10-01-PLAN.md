<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| extract request -> service target selection | Untrusted `segmentIds` and approval-reset flags determine which stored segments can be retried. |
| service -> repository attempt persistence | Extraction results and failures cross from transient LLM responses into durable retry lineage. |
| repository snapshot -> API response | Stored attempt history and progress counts become writer-visible operational state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | approved-segment retry flow | mitigate | Default approved-target retries to 409 and require `allowApprovalReset=true` before clearing approval or replacing current candidates. |
| T-10-02 | Repudiation | extraction lineage | mitigate | Append one `ingestion_segment_attempts` row per targeted attempt and never store retry history only in mutable segment JSON. |
| T-10-03 | Integrity | mixed extraction state | mitigate | Compute and serialize `progressSummary` plus per-segment `lastAttemptStatus`, `lastFailureSummary`, and `stale` fields from repository truth. |
| T-10-04 | Tampering | targeted extract API | mitigate | Validate `segmentIds` structurally in Zod and semantically in the service by checking existence, duplicates, ownership, and approved-target conflicts. |
</threat_model>
